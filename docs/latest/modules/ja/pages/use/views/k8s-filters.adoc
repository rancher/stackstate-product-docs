== 概要

ビューUIの右上隅にある*フィルター*メニューを使用すると、ビューに表示されるコンポーネント（トポロジー）やイベントを絞り込むことができます。適用されると、フィルターはビュー内のすべての視点の内容に影響を与えます。

image::k8s/k8s-filters.png[ビューフィルター]

== トポロジーフィルター

トポロジーフィルターを使用して、利用可能な視点のいずれかに表示されるトポロジーコンポーネントのサブセットを選択できます。イベントフィルターはすべてのビュータイプで同じですが、トポロジーフィルターはビューのタイプによって異なります。詳細はこちら：

* xref:/use/views/k8s-filters.adoc#_topology_filters_on_kubernetes_views[Kubernetesビューのトポロジーフィルター]
* xref:/use/views/k8s-filters.adoc#_topology_filters_on_other_view_types[他のビュータイプのトポロジーフィルター]

=== Kubernetesビューのトポロジーフィルター

xref:/use/views/k8s-views.adoc[Kubernetesビュー]では、トポロジーフィルターは、すべてのKubernetesビューにわたって持続する基本フィルターの小さなセットに制限されています：`clusters`および/または`namespaces`。Kubernetesビューの持続的なトポロジーフィルターは、UIの通常の`filters`メニューの外に配置されており、互いに依存しません（例：クラスターを選択しても、*自動的に*そのクラスターのネームスペースのみを反映するように`namespaces`フィルターが更新されることはありません）。

image::k8s/k8s-filters-kube-topology.png[Kubernetesトポロジーフィルター]

=== 他のビュータイプのトポロジーフィルター

他のビュータイプでは、基本または高度なトポロジーフィルターを使用してトポロジーをブラウズできます。詳細については、次の情報をお読みください。

* xref:/use/views/k8s-filters.adoc#_basic_topology_filters[基本トポロジーフィルター]
* xref:/use/views/k8s-filters.adoc#_advanced_topology_filters[高度なトポロジーフィルター]
* xref:/use/views/k8s-filters.adoc#_topology_filtering_limits[トポロジーフィルタリングの制限]

==== 基本トポロジーフィルター

トポロジーをフィルタリングする主な方法は、利用可能な基本フィルターを使用することです。フィルターを設定すると、開いている視点はフィルターに一致するトポロジーのサブセットの可視化またはデータのみを表示するように更新されます。複数のフィルターを設定すると、検索がさらに絞り込まれます。各フィルターに対して複数の値を設定することで、検索を拡張できます。

|===
| フィルタ | 説明

| レイヤー、ドメイン、環境、およびコンポーネントタイプ
| コンポーネントがインポートまたは作成される際に含まれるコンポーネントの詳細でフィルターします。

| コンポーネントの健康状態
| 関連する健康チェックによって報告された名前付きの健康状態を持つコンポーネントのみを含めます。

| コンポーネントラベル
| 特定のラベルを持つコンポーネントのみを含めます。

| コンポーネントを含める
| ここに名前が付けられたコンポーネントは、他のフィルターから返されたコンポーネント*に加えて*トポロジーに含まれます。
|===

基本フィルターを使用してトポロジーをフィルターするには、UIの右上隅にある*フィルター*メニューをクリックし、*トポロジー*の垂直タブの下にある*基本に切り替え*を選択します。

image::k8s/k8s-filters-basic-topology.png[基本トポロジーフィルター]

==== 高度なトポロジーフィルター

組み込みのxref:/develop/reference/k8sTs-stql_reference.adoc[SUSE Observability Query Language (STQL)]を使用して、トポロジーの特定の領域にズームインする高度なトポロジーフィルターを構築できます。

STQLクエリを使用してトポロジーをフィルターするには、UIの右上隅にある*フィルター*メニューをクリックし、*トポロジー*の垂直タブの下にある*STQLに切り替え*を選択します。

以下のSTQLクエリの例は、次の条件に一致するコンポーネントを返します。

* *ドメイン*の`security check`
* AND *健康*状態が`Clear`または`Deviating`である
* OR 名前が`ai_engine`の*コンポーネント*である

[,yaml]
----
(domain IN ("security check") AND healthstate IN ("CLEAR", "DEVIATING")) OR name IN ("ai_engine")
----

image::k8s/k8s-filters-advanced-topology.png[高度なトポロジーフィルター]

==== 基本フィルターと高度なフィルターの互換性

*トポロジー*の垂直タブにある*フィルター*メニューの下で、*基本に切り替え*または*STQLに切り替え*を選択することで、基本フィルターと高度なフィルターの間を切り替えることができます。

基本フィルターから高度なフィルターに切り替えることは常に可能です。選択された基本フィルターは、直接STQLクエリに変換されます。簡単なクエリの場合、高度なフィルターから基本フィルターに切り替えることも可能ですが、一部の高度なクエリは基本フィルターと互換性がありません。

➡️ xref:/develop/reference/k8sTs-stql_reference.adoc#_compatibility_basic_and_advanced_filters[基本フィルターと高度なトポロジーフィルターの互換性について詳しく学ぶ]

==== その他のフィルター

以下に示す高度なフィルターは基本フィルターと互換性がありますが、基本フィルターとして設定または調整することはできません。

* *withNeighborsOf* - 高度なフィルターにxref:/develop/reference/k8sTs-stql_reference.adoc#_withneighborsof[withNeighborsOf]関数が含まれている場合、隣接するコンポーネントの数が*その他のフィルター*ボックスに表示されます。基本フィルターと互換性を持つためには、`withNeighborsOf`関数は他のフィルターと`OR`演算子を使用して結合されなければなりません。
* *identifier* - 高度なフィルターがxref:/develop/reference/k8sTs-stql_reference.adoc#_filters[識別子]によってコンポーネントを選択する場合、照会されたコンポーネント識別子の数が*その他のフィルター*ボックスに報告されます。基本フィルターと互換性を持つためには、`identifier`フィルターを指定し、`OR identifier IN (...)`演算子を使用して他のフィルターと結合する必要があります。

基本トポロジーフィルターの*その他のフィルター*ボックスには、これらのすべての高度なフィルターと影響を受けるコンポーネントの数がリストされています。

[NOTE]
====
*その他のフィルター*ボックスは、設定されていて基本フィルターと互換性のある高度なフィルターの詳細のみを提供します。
====


==== トポロジーフィルタリングの制限

パフォーマンスを最適化するために、トポロジービジュアライゼーションを生成するために読み込むことができる要素の量に制限が設けられています。フィルタリング制限のデフォルト値は10000要素です。xref:/use/views/k8s-filters.adoc#_basic_topology_filters[基本フィルター]またはxref:/use/views/k8s-filters.adoc#_advanced_topology_filters[高度なフィルタークエリ]がフィルタリング制限を超えると、画面にメッセージが表示され、トポロジービジュアライゼーションは表示されません。

フィルタリング制限は、*読み込まれる必要がある*要素の総量に適用され、最終的に表示される要素の量には適用されないことに注意してください。

以下の例では、最初にトポロジー内のすべてのコンポーネントの隣接ノードを読み込み、その後`applications`階層内に属するものだけを表示します。これは、トポロジー内のすべてのコンポーネントを読み込む必要があるため、フィルタリング制限エラーで失敗する可能性があります。

[,text]
----
withNeighborsOf(direction = "both", components = (name = "*"), levels = "15")
   AND layer = "applications"
----

このトポロジビジュアライゼーションを成功裏に生成するには、読み込むコンポーネントの数を設定されたフィルタリング制限以下に保つようにクエリを再記述するか、フィルタリング制限を増加させる必要があります。〈p-1 /〉階層内のコンポーネントのみをフィルタリングすることにより、すべてのコンポーネントを最初に読み込むことなく、上記のクエリと同じコンポーネントを表示します。したがって、このクエリはフィルタリング制限エラーが発生する可能性が低くなります。

[,yaml]
----
layer = "applications"
----

== イベントのフィルタ処理:

SUSE Observability UIの左側にある*ビューフィルター*パネルを使用して、xref:/use/views/k8s-events-perspective.adoc[Events Perspective]に表示されるイベントをフィルタリングできます。それらは、右側のパネルの*Event*リストおよび詳細タブ - *View summary*、*Component details*、*Direct relation details*にも含まれています。

以下のイベントフィルターが利用可能です。

|===
| フィルタ | 説明

| *カテゴリ*
| 1つ以上のxref:/use/views/k8s-events-perspective.adoc#_event_category[categories]からのイベントのみを表示します。

| *タイプ*
| *Type*フィルターボックスをクリックすると、現在フィルタリングされているコンポーネントに対して生成されたすべてのイベントタイプのリストが表示されます。表示されるイベントを絞り込むために、1つ以上のイベントタイプを選択できます。

| *ソース*
| イベントはSUSE Observabilityによって生成されるか、KubernetesやServiceNowなどの外部ソースシステムから統合によって取得されることがあります。*Source*フィルタボックスをクリックすると、現在フィルタリングされているコンポーネントに対して生成されたイベントのすべてのソースシステムのリストが表示されます。1つ以上のソースシステムを選択して、これらのイベントのみを表示します。

| *タグ*
| 関連するイベントプロパティは、外部システムからイベントが取得されるときにタグとして追加されます。例えば`status:open`または`status:production`。これにより、特定の問題や環境に関連するイベントを特定するのに役立ちます。
|===
