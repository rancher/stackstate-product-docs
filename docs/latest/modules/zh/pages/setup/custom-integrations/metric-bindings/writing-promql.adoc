== 准则

当 {stackstate-product-name} 在图表中显示数据时，几乎总是需要更改存储数据的分辨率，以使其适合图表的可用空间。为了获得尽可能代表性的图表，请遵循以下指南：

* 不要查询原始指标，而是始终在时间上进行聚合（使用 `*_over_time` 或 `rate` 函数）。
* 将 `${__interval}` 参数用作时间聚合的范围，它会随着图表的分辨率自动调整。
* 将 `${__rate_interval}` 参数用作 `rate` 聚合的范围，它也会随着图表的分辨率自动调整，但会考虑 `rate` 的特定行为。
* 通过对不同时间序列进行聚合，将项目指标限制在实际使用的标签上。

应用聚合通常意味着在指标中强调某些模式而牺牲其他模式。例如，对于较大的时间窗口，`max_over_time` 会显示所有峰值，但不会显示所有谷值。而 `min_over_time` 则完全相反，`avg_over_time` 会平滑处理峰值和谷值。为了展示这种行为，这里是一个使用 Pod CPU 使用率的示例指标绑定。要自己尝试，请将其复制到 YAML 文件中，并在您自己的 {stackstate-product-name} 中使用 xref:/setup/custom-integrations/metric-bindings/index.adoc#_create_or_update_the_metric_binding_in_stackstate[CLI 来应用它]（您可以稍后去除）。

将多个时间序列投影到其标签的子集上，将在无关细节上不同的时间序列折叠在一起。 在创建指标绑定时，仅图例中使用的标签是相关的。 同样，在创建监控时，仅查询中需要映射到组件（监控状态）的标签应被返回。

----
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: CPU Usage (different aggregations and intervals)
  priority: HIGH
  identifier: urn:stackpack:my-stackpack:metric-binding:pod-cpu-usage-a
  queries:
    - expression: sum(max_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: max_over_time dynamic interval
    - expression: sum(min_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: min_over_time dynamic interval
    - expression: sum(avg_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: avg_over_time dynamic interval
    - expression: sum(last_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: last_over_time dynamic interval
    - expression: sum(max_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: max_over_time 1m interval
    - expression: sum(min_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: min_over_time 1m interval
    - expression: sum(avg_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: avg_over_time 1m interval
    - expression: sum(last_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: last_over_time 1m interval
  scope: (label = "stackpack:kubernetes" and type = "pod")
----

应用后，打开 {stackstate-product-name} 中 Pod 的指标视角（最好选择一个处理器使用率出现峰值和谷值的 Pod）。使用图表右上角的图标放大图表，以获得更好的视图。现在，您还可以更改时间窗口，以查看不同聚合的效果（例如 30 分钟与 24 小时）。

[CAUTION]
====
当指标绑定未指定聚合时，{stackstate-product-name} 将自动使用 `last_over_time` 聚合来减少图表的数据点数量。另请参见 xref:/setup/custom-integrations/metric-bindings/writing-promql.adoc#_why_is_aggregation_necessary[为什么聚合是必要的？] 以获取解释。
====

== 为什么聚合是必要的？

首先，您为什么应该使用聚合？从度量存储中检索的数据点数量不能超过图表所能容纳的数量。因此，{stackstate-product-name} 会自动确定两个数据点之间所需的步长，以获得良好的结果。对于短时间窗口（例如仅显示一小时数据的图表），这会导致较小的步长（大约 10 秒）。度量通常每 30 秒收集一次，因此对于 10 秒的步长，相同的值将在三个步长内重复，然后才会更改为下一个值。缩小到一周的时间窗口将需要更大的步长（大约一小时，具体取决于屏幕上图表的确切大小）。

当步长大于收集的数据点的分辨率时，需要决定如何将一小时时间范围内的数据点汇总为一个值。当查询中已经指定了时间上的聚合时，将使用该聚合来进行汇总。然而，如果没有指定聚合，或者当聚合间隔小于步长时，将使用 `last_over_time` 聚合，间隔为 `step` 大小。结果是每小时仅使用最后一个数据点来“汇总”该小时内的所有数据点。

总之，当执行针对一周时间范围且步长为一小时的 PromQL 查询时，该查询：

----
container_cpu_usage /1000000000
----

会自动转换为：

----
last_over_time(container_cpu_usage[1h]) /1000000000
----

在 https://observability.suse.com/#/metrics?alias=Pod%20%24%7Bpod_name%7D&promql=last_over_time%28container_cpu_usage%7Bnamespace%3D%22sock-shop%22%2Cpod_name%3D~%22carts.%2A%22%7D%5B1h%5D%29%20%2F1000000000&timeRange=LAST_7_DAYS[{stackstate-product-name} 游乐场] 上自己试试。

image::k8s/k8s-metric-queries-for-chart-last-over-time.png[随时间变化的最后值]
image::k8s/k8s-metric-queries-for-chart-max-over-time-fixed-range.png[固定范围的最大值随时间变化]
image::k8s/k8s-metric-queries-for-chart-max-over-time-interval.png[自动范围的最大值随时间变化]

这种行为通常不是预期的，最好自己决定需要什么样的聚合。使用不同的聚合函数可以强调某些行为（代价是隐藏其他行为）。更重要的是查看峰值、谷值、平滑图表等吗？那么使用 `${__interval}` 参数作为范围，因为它会自动替换为查询中使用的 `step` 大小。结果是步长内的所有数据点都被使用。

image::k8s/k8s-metric-queries-small-range.png[固定范围, shorter than the data resolution]
image::k8s/k8s-metric-queries-interval-for-range.png[自动范围, based on step but with a lower limit]

`${__interval}` 参数防止了另一个问题。当`step`的大小以及`${__interval}`的值缩小到小于存储的度量数据的分辨率时，这将导致图表中出现空白。

因此，`${__interval}` 永远不会缩小到小于 {stackstate-product-name} 代理的默认抓取间隔的两倍（默认抓取间隔为 30 秒）。

最后，`rate()` 函数至少需要在该区间内具备两个数据点，才能计算出速率。如果数据点不足两个，速率将没有值。因此，`${__rate_interval}` 保证始终至少为抓取间隔的 4 倍。这保证了速率图表中不会出现意外的空白或其他奇怪的行为，除非数据缺失。
